# Experiment 5: Migrate Badge to Tailwind utilities

## Description

With the candidate fixture loading real Tailwind v4 application-wide
(Experiment 4) against the theme/token contract (Experiment 2), this
experiment performs the first true component migration: Badge moves from
bespoke `radcn-badge*` CSS to shadcn-faithful Tailwind utility classes. Badge
is the representative because it is self-contained, has the full variant set
(default/secondary/destructive/outline/ghost/link), and is consumed only
through its `<Badge>` JSX API — no fixture references `radcn-badge` as a raw
class string (verified), so removing the bespoke CSS cannot strand other
markup.

This experiment also applies the maintainer's decision recorded in the issue
README: per-component `--radcn-*` custom-token override hooks are dropped for
shadcn parity. Today Badge reads `var(--radcn-badge-bg, ...)`/`var(--radcn-badge-fg, ...)`,
and the `custom-class` fixture scenario overrides via
`.radcn-fixture-custom-badge { --radcn-badge-bg: #4f46e5 }`. After migration
Badge emits only utilities; the override moves to the shadcn form — a class
that sets the color directly.

shadcn v4 Badge (vendored, `registry/new-york-v4/ui/badge.tsx`) is the
mapping source, reproduced VERBATIM (not a simplified subset) so the migration
is faithful — including the transition, focus-visible, aria-invalid, and SVG
utilities and the per-variant hover states. RadCN adds nothing visual beyond
shadcn's strings; it only keeps the `data-radcn-badge`/`data-variant`
non-visual hooks and renders an `<a>` when `href` is set (which is exactly
what makes shadcn's `[a&]:` anchor-scoped hover variants meaningful).

These strings are transcribed exactly from the vendor source (token order
preserved); implementation must copy them directly from
`vendor/shadcn-ui/apps/v4/registry/new-york-v4/ui/badge.tsx` rather than
retype them, so the result is byte-for-byte faithful.

- base: `inline-flex w-fit shrink-0 items-center justify-center gap-1
  overflow-hidden rounded-full border border-transparent px-2 py-0.5 text-xs
  font-medium whitespace-nowrap transition-[color,box-shadow]
  focus-visible:border-ring focus-visible:ring-[3px] focus-visible:ring-ring/50
  aria-invalid:border-destructive aria-invalid:ring-destructive/20
  dark:aria-invalid:ring-destructive/40 [&>svg]:pointer-events-none
  [&>svg]:size-3`
- default: `bg-primary text-primary-foreground [a&]:hover:bg-primary/90`
- secondary: `bg-secondary text-secondary-foreground [a&]:hover:bg-secondary/90`
- destructive: `bg-destructive text-white focus-visible:ring-destructive/20
  dark:bg-destructive/60 dark:focus-visible:ring-destructive/40
  [a&]:hover:bg-destructive/90`
- outline: `border-border text-foreground [a&]:hover:bg-accent
  [a&]:hover:text-accent-foreground`
- ghost: `[a&]:hover:bg-accent [a&]:hover:text-accent-foreground`
- link: `text-primary underline-offset-4 [a&]:hover:underline`

Utility resolution:

- Color-token utilities (`bg-primary`, `text-primary-foreground`,
  `bg-secondary`, `text-secondary-foreground`, `bg-destructive`,
  `border-border`, `text-foreground`, `text-primary`, `border-ring`,
  `ring-ring`, `ring-destructive`, `bg-accent`, `text-accent-foreground`)
  resolve through the Experiment 2 `@theme inline` contract; the matching
  `--color-*` tokens are all present in
  `radcn/packages/radcn/src/styles/theme.css`. `text-white` is a Tailwind
  built-in color.
- Structural/scale utilities (`inline-flex`, `w-fit`, `shrink-0`, `gap-1`,
  `overflow-hidden`, `rounded-full`, `border`, `border-transparent`, `px-2`,
  `py-0.5`, `text-xs`, `font-medium`, `whitespace-nowrap`, `ring-[3px]`,
  `size-3`, `underline-offset-4`, `underline`, `pointer-events-none`,
  `transition-[color,box-shadow]`) come from Tailwind v4's default theme,
  which `@import 'tailwindcss/theme'` (the first import in the fixture's
  `tailwind.css`) provides; the Experiment 2 contract only adds/overrides
  color and `--radius-*` tokens, leaving the default type/spacing/radius-full
  scales intact.
- Opacity modifiers (`/90`, `/50`, `/20`, `/40`, `/60`) and the `dark:`
  variant resolve via Tailwind v4 color-mix and the Experiment 2
  `@custom-variant dark` declaration.
- All are generated by `styles:build` because Experiment 4 added `@source`
  scanning of the package source; verification step 1 confirms the variant
  utilities actually appear in the generated CSS (and fails loudly if any
  utility or modifier does not generate).

Scope limits: Badge only. No preflight change (badge utilities do not depend
on the base reset). No change to other components. The `data-radcn-badge` and
`data-variant` attributes stay — they are non-visual hooks the tests and any
consumers use as selectors, exactly the "data attributes may remain" allowance
in the issue Decisions. The docs app is untouched (it does not yet load
Tailwind; that is a later experiment).

## Why the suite stays green

The Badge tests (in `tests/static-display.spec.ts`) assert text, attributes,
class presence, and only two computed styles, both of which survive:

- The `demo` "Verified" badge asserts `background-color: rgb(37, 99, 235)`
  (`#2563eb`). That comes from an inline `style="background:#2563eb;color:white"`
  on the fixture element — inline styles win over any utility, so it is
  unaffected by the migration.
- The `demo` count badges assert `border-radius: 999px` from inline
  `style="border-radius:999px;..."` — likewise inline, unaffected.
- The `custom-class` badge asserts `background-color: rgb(79, 70, 229)`
  (`#4f46e5`). Today that comes from `.radcn-fixture-custom-badge` setting
  `--radcn-badge-bg`. After migration Badge ignores that variable, so the
  fixture class is changed to set the color directly (the shadcn "override via
  class" form), using LONGHAND properties to override the badge's longhand
  utilities unambiguously:
  `.radcn-fixture-custom-badge { background-color: #4f46e5; color: #ffffff; }`.
  Cascade reasoning (and its limits): the `custom-class` scenario renders the
  DEFAULT variant, whose utilities set `background-color`
  (`bg-primary` → `background-color: var(--primary)`) and `color`. The
  override must therefore win those exact properties. It does, by document
  order: `.radcn-fixture-custom-badge` lives in `radcnStyles`, which the shell
  injects AFTER the Tailwind `<link>` (Experiment 4 ordering); both are
  single-class specificity, so the later rule wins. The asserted `#4f46e5`
  computes and `toHaveClass(/radcn-fixture-custom-badge/)` still holds. This
  override is property-level and order-dependent by design: it works because it
  sets the same properties the variant utilities set and is ordered after them.
  That is acceptable for a fixture override hook; it is not a general styling
  mechanism (the issue's decision is that real overrides happen at the call
  site via the `class` prop). The full-suite gate confirms the computed result.

No base-badge geometry (padding/font/default background/border-radius) is
asserted anywhere, so switching the base to shadcn's utilities (e.g.
`rounded-full`) is safe. The full fixture suite is the gate.

## Changes

- `radcn/packages/radcn/src/components/badge.tsx`:
  - Replace `classes('radcn-badge', \`radcn-badge--${variant}\`, className)`
    with the verbatim shadcn v4 base utility string (the "base:" list above)
    plus the matching per-variant string (a `variant`→string record keyed by
    `default`/`secondary`/`destructive`/`outline`/`ghost`/`link`) plus the
    author `className`, merged with the existing `classes(...)` helper. Keep
    `data-radcn-badge`, `data-variant`, the `href` → `<a>` branch, and
    `style`/`class` passthrough unchanged. No `radcn-badge*` class is emitted
    anymore.
- `radcn/packages/radcn/src/styles/tokens.css`:
  - Remove the `.radcn-badge`, `.radcn-badge--secondary`,
    `.radcn-badge--destructive`, `.radcn-badge--outline`, `.radcn-badge--ghost`,
    and `.radcn-badge--link` rules (now expressed as utilities in the
    component).
  - Change `.radcn-fixture-custom-badge` from setting `--radcn-badge-bg`/
    `--radcn-badge-fg` to setting `background: #4f46e5; color: #ffffff;`
    directly (shadcn-style class override; preserves the `custom-class` test's
    computed color and class-presence assertions).
- `radcn/packages/radcn/src/styles/index.ts`:
  - Regenerate from the edited `tokens.css` with the exact formula
    `export const radcnStyles = ${JSON.stringify(tokensCss)}\n` so the two
    files stay byte-identical (per Experiment 3's recorded learning).

No fixture markup change, no test change: the migration is constructed so the
existing Badge assertions remain valid.

Expected git status for the result commit: `badge.tsx`, `tokens.css`,
`index.ts`, this experiment file, and the issue README index line. The
generated `tailwind.generated.css` stays untracked.

## Conclusion

The first component migration is proven end to end: Badge now renders entirely
from shadcn v4 Tailwind utility classes resolved through the RadCN theme
contract, its bespoke `radcn-badge*` CSS is deleted, and its non-visual
`data-radcn-badge`/`data-variant` hooks remain. The maintainer's
custom-token-drop decision is realized in practice — the `custom-class`
scenario now overrides via a direct class rule (the shadcn "override via class"
form) rather than a `--radcn-badge-bg` hook — with zero test churn and the full
suite green.

Learnings for later experiments (also copied to the issue README Learnings
digest):

- The per-component migration recipe that works: (a) copy the shadcn v4
  utility strings verbatim from the vendored registry into the component as a
  `base` string + a `Record<Variant, string>`; (b) keep `data-*` hooks and the
  href→`<a>` rendering; (c) delete the component's `radcn-*` visual rules from
  `tokens.css`; (d) regenerate `index.ts`; (e) translate any fixture
  custom-token override to a direct class rule (longhand properties, which win
  the cascade tie because `radcnStyles` is ordered after the Tailwind link).
- Fixture inline styles and direct fixture-class rules both still win over the
  component's utilities, so scenarios pinned by inline styles need no change;
  only `--radcn-*`-hook-driven scenarios need translating.
- shadcn variant strings reference `accent`/`ring` tokens and opacity
  modifiers (`/90`, `/50`) and `dark:` states; all generate correctly against
  the Experiment 2 contract + Tailwind defaults, so later components can use
  the full shadcn strings without trimming.

The next experiment migrates the next component cluster (Button is the natural
choice, but it is referenced by raw `radcn-button` class strings across many
fixtures — so its experiment must either migrate those call sites together or
keep `radcn-button` as a compatibility alias; that decision is the next
experiment's design problem).
